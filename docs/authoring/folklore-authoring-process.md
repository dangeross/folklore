# FOAKLOAR — Authoring Process

*Attach this doc + `folklore-authoring-guide.md` + `folklore-design.md` to start a session.*  
*Give your theme in the first message. That is all that is required.*

The goal is a world that feels genuinely surprising — one that couldn't have been generated by following a template. The spec is the only constraint. Everything else is fair game.

---

## Step 1 — Treatment

Write a treatment of 3–5 sentences from the theme alone.

The treatment is not a story summary. It is a statement of what makes this world worth playing — the central strangeness, the thing the player will feel that they haven't felt in another game. It should be specific enough to be wrong. A treatment that could apply to ten different worlds isn't doing its job.

Do not default to: a hero with a clear goal, a villain, a linear revelation, a tidy moral. These are available options, not defaults.

Ask: what assumption about how this kind of story works can be inverted? What if the player's goal is unclear, or wrong, or shifts? What if the win state is morally ambiguous? What if the world existed for a long time before the player arrived and will continue after they leave?

End the treatment with one sentence that states what the win state *feels like* — not what it is mechanically, but what the player carries with them when it's over.

**Present the treatment and wait. The user will approve, redirect, or say nothing and let you continue.**

---

## Step 2 — Design

From the treatment, make all structural decisions internally. Do not present a brief for approval — commit to a direction and build.

Ask yourself, but do not report unless interesting:

- What is the win state and how does the player unlock it cryptographically?
- What is the knowledge or item chain — and does it have to be linear, or can it branch, or have a false path?
- Does the player need items at all, or can progression be driven by relationship state, observation, or payment?
- How many places, and do they have to be navigated in order?
- What is the tone — not just mood, but voice? How does the world speak?

The spec supports: non-linear chains, multiple valid paths to the win state, worlds where the player's role is ambiguous, NPCs who are not helpers, places that feel wrong, win states that cost something. Use what serves the treatment.

**One hard rule:** for worlds of 10 or more places, present the map — place names, exits, and portal connections only — before writing features, items, and NPCs. Map errors are expensive to fix late. Everything else can be written in one pass.

---

## Step 3 — Write events

Write the full event JSON in authoring order:

```
world → places → portals → items → features → clues →
puzzles → NPCs → dialogue → consequences → sounds
```

**File format:** use the `answers` + `events` structure. The `answers` map holds plaintext puzzle answers keyed by puzzle `d`-tag. NIP-44 sealed place `content` fields hold the plaintext win prose — the publishing tool encrypts them.

**Hashes:** compute real SHA256 hashes for all `answer-hash` tags using `SHA256(answer + salt)`. Never output placeholder strings.

**Sound:** score the world against the treatment, not the map. Sound should make the player feel what the treatment promised — if the treatment is about dread and false hope, the sound should carry that. Follow the layering budget from the spec: 1 ambient + 1 optional layer per place, 3–4 active layers maximum.

---

## Step 4 — Validate

Run the full validation checklist from the authoring guide. Fix all errors and warnings.

Then POST the clean JSON to the validation API:

```
POST https://flklr.com/api/validate
Content-Type: application/json
```

Apply all issues returned. Re-validate locally after API fixes. Deliver only when both are clean.

**Final deliverable is valid JSON with no JS comments** — strip all `// comments` before delivering.

---

## What the spec does not prevent

If something feels like it might be too strange for FOAKLOAR, check the spec before pulling back. The schema supports:

- Worlds where the player's stated goal is not the real goal
- NPCs who actively work against the player, steal items, or provide false information
- Multiple valid paths to the win state with meaningfully different outcomes
- Win states that are sealed behind moral choices, not just puzzle solutions
- Places that can only be reached by failing something first
- Clues that are comprehensible in hindsight but deliberately misleading in foresight
- Worlds with no obvious protagonist arc — the player as witness rather than hero
- Payment gates that create genuine decisions about cost versus progress
- Dialogue trees where the wrong answer is more interesting than the right one

The spec is the floor, not the ceiling.
