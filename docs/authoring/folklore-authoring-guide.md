# FOAKLOAR — World Authoring Guide

*How to design, write, and publish a world*

---

## What You Are Building

A FOAKLOAR world is a **directed graph of events** stored on NOSTR relays (kind 30078). Places are nodes. Portals are edges. Everything else — features, items, NPCs, puzzles, clues — hangs off the nodes as content and behaviour.

The player navigates the graph by following portals. They interact with content to change state. State changes open new paths, reveal new content, and eventually reach the win state. The story emerges from the player's movement through the world — from what they find, earn, and unlock.

The spec is the only constraint. The schema supports non-linear structures, unreliable worlds, ambiguous goals, NPCs who work against the player, win states that cost something, and worlds where the player's role is never named. Use whatever the world needs.

## D-Tag Naming Convention

Every event has a deterministic `d`-tag that encodes its world, type, and identity.

| Event | D-tag format | Example |
|-------|-------------|---------|
| World | `<slug>:world` | `the-lake:world` |
| All others | `<slug>:<type>:<name>` | `the-lake:place:clearing` |

- `<slug>` is the world's `t` tag value — lowercase, hyphenated
- `<type>` matches the `type` tag: `place`, `portal`, `item`, `feature`, `npc`, `dialogue`, `puzzle`, `quest`, `recipe`, `clue`, `consequence`, `sound`, `payment`, `vouch`
- `<name>` is a short, descriptive, hyphenated identifier

Every event also carries three identity tags (auto-generated by the builder):

```
["d", "<d-tag>"], ["t", "<slug>"], ["type", "<event-type>"]
```

---

## Event Ref Format

Cross-event references use the NOSTR `a`-tag format:

```
30078:<PUBKEY>:<d-tag>
```

During authoring, use `<PUBKEY>` as a literal placeholder. The publishing tool substitutes your real pubkey at sign time.

**Example:** `30078:<PUBKEY>:the-lake:item:iron-key`

All `requires`, `exit` (on portals), `item`, `feature`, `npc`, `clue`, `dialogue`, `involves`, `on-*` external targets, and `contains` tags use this format.

## World Design Process

### Before you write a single event

Answer three questions:

1. **What does winning feel like?** Not the mechanism — the feeling. This is the thesis of the world. Every place, item, and clue exists to earn that feeling.

2. **What is the progression chain?** The sequence of things the player must acquire or discover to reach the win state. Work backwards from the end. The chain can branch, converge, or include false paths — but every link must be reachable, and no link should require itself.

3. **What is the voice?** One sentence, present tense, describing how the world sounds. Every content field and transition text will be checked against it.

### Draw the map

A FOAKLOAR map is a graph. Boxes for places, lines for portals, annotations for gates. The map reveals structural problems invisible in prose.

A good map has:
- A clear critical path from start to win state
- At least one place that rewards curiosity without being required
- At least one intentional dangling exit — a signal to collaborators or a door that never opens
- A sense of depth as the player goes deeper

### Write prose first

Write the `content` field for every place before writing any event JSON. Read it in sequence along the critical path. Does the voice stay consistent? Does tension build or release at the right moments? Fix prose before touching events.

### Collaboration mode

Choose before publishing — it is the first trust decision.

| Mode | Who can extend | Use for |
|------|---------------|---------|
| `closed` | Only you | Canonical story worlds |
| `vouched` | You + listed collaborators | Curated shared worlds |
| `open` | Anyone | Community worlds |

## Content Field vs Tags

| Goes in `content` | Goes in tags |
|-------------------|-------------|
| Prose descriptions of places, items, features, NPCs, clues | Structural data: exits, nouns, verbs, states, triggers |
| Dialogue spoken text | Dialogue options and branching refs |
| Quest summary text | Quest type, involves, requires, on-complete |
| Consequence narration | Respawn, clears, give-item, consume-item |
| Win state prose (NIP-44 encrypted) | Puzzle answer-hash, salt |

**Rule:** If a human reads it, it goes in `content`. If the engine evaluates it, it goes in tags.

Content is required on: `place`, `item`, `feature`, `npc`, `clue`. Content is optional/empty on: `portal`, `consequence`, `sound`, `payment`, `vouch`. Dialogue `content` is the NPC's spoken text.

---

## Writing Guidelines

- **Present tense.** "The cave opens into darkness" not "the cave opened."
- **Sensory detail first.** What does the player see, hear, smell?
- **Don't explain the puzzle.** The place describes the environment. Features describe interactable things. Clues carry information. Keep them separate.
- **Transition text is feedback, not description.** One sentence maximum. It fires inline between input and next output.
- **Clues point, they don't instruct.** The player should understand a clue *after* following it, not before.
- **Dialogue is character voice, not exposition.** Information should be inferrable, not delivered directly.

---

## Event Authoring Order

Write and publish events in this order (dependencies flow downward):

1. World event
2. Place events (with exit slots declared)
3. Portal events (fill the exit slots)
4. Item events
5. Feature events
6. Clue events
7. Puzzle and payment events
8. NPC events
9. Dialogue events (leaf nodes before root)
10. Consequence events
11. Sound events
12. Quest events

## Common Patterns

### Gated door

A portal that requires an item or state before the player can pass through.

- Portal has `requires` tag referencing an item or feature state
- Player acquires the item or changes the feature state elsewhere
- Portal becomes passable

### State machine

An entity (item, feature) with multiple states and transitions between them.

- `state` sets the initial state
- `transition` tags define from/to pairs with feedback text
- `on-interact` tags bind player verbs to `set-state` actions — state guard (position 2) can gate which state the entity must be in for the action to fire (blank = any state)

### Fetch quest

A quest that tracks collection of items or completion of objectives.

- `quest` event with `quest-type` and `involves` tags
- `requires` tags on the quest define completion conditions
- `on-complete` fires rewards when all conditions pass

### Environmental puzzle

No puzzle event needed — just `requires` on portals and `set-state` on features. The player reads the environment and figures out the interaction chain.

### NPC broker

An NPC whose dialogue changes based on world state. Multiple `dialogue` tags with `requires` — last passing entry wins. The player who has explored more gets richer conversation.

### Converging paths

Two routes to the same destination with different costs. One requires puzzle-solving (free, takes time). The other requires payment (fast, costs sats).

### Branching via state

There is no special branching mechanic. Forks are expressed through `set-state` on any trigger (dialogue choice, item used, puzzle solved). The state gates future content wherever `requires` is declared.

### Drop receptacle (on-drop on a feature)

A feature that reacts when a specific item is dropped into or onto it — a well, a slot, a bowl, an altar.

- Feature carries `on-drop` tags with the item ref in position 1 and a state guard in position 2 (blank = any state)
- Player uses `drop X in/on/into Y` — the client resolves Y to the feature and fires matching `on-drop` tags
- Plain `drop X` (no feature target) does not trigger feature `on-drop` — item falls to the floor silently
- Use `set-state` with ext-ref to change the dropped item's own state (mark it as deposited, consumed, placed)
- **All matching tags fire in declaration order** — the same semantics as `on-interact`. Multiple `on-drop` tags with the same item-ref + state-guard key all execute sequentially. Put side effects that depend on earlier ones (e.g. counter checks after a `set-state`) later in the tag list.

### Drop-triggered place effect (on-drop on a place)

A place that reacts to any drop — or to a specific item being dropped anywhere in the room. Useful for pressure-sensitive floors, scent-based traps, or environmental puzzles where placement matters.

- Place carries `on-drop` tags
- Fires on plain `drop X` — no explicit feature target needed
- Item-ref (position 1) blank = any item triggers the handler; specific ref = only that item
- **All matching tags fire in declaration order** — same semantics as `on-interact` and feature `on-drop`

### Inline actions vs consequences

One action from one trigger: use inline `on-*` tags. Multiple actions firing together, or the same outcome from multiple triggers: use a `consequence` event.

## NIP-44 Win State

For sealed win content (decrypted only when the player solves a puzzle):

1. The puzzle `answer-hash` verifies the player's input: `SHA256(answer + salt)`
2. The same answer is key material for NIP-44 decryption of the win place content
3. The publishing tool handles encryption — you provide plaintext in the `content` field

**Hash format:** The hash is computed as `SHA256(answer + salt)` where `+` means string concatenation (no separator). Both `answer` and `salt` are treated as literal strings. The result is a lowercase hex digest (64 characters).

**Example:**
```
answer: "bottle"
salt:   "the-lake:puzzle:chapel-riddle:v1"
input:  "bottlethe-lake:puzzle:chapel-riddle:v1"
hash:   "cc421c90e82bd9b59dd998b616f8cb2f35270149388e2e042b9151d64a4e34b8"
```

**LLM authors must:**
- Compute the real SHA-256 hex digest for `answer-hash` — a placeholder publishes a permanently unsolvable puzzle
- The `salt` tag should be unique per puzzle (convention: `<world-slug>:puzzle:<name>:v1`)
- Write the plaintext win prose in the `content` field, clearly labelled — the publishing tool encrypts it

## Validation

Before publishing, validate your world JSON against the API.

**Endpoint:** `POST https://flklr.com/api/validate`

```bash
curl -X POST https://flklr.com/api/validate \
  -H "Content-Type: application/json" \
  -d @my-world-events.json
```

**For expansions** (adding to an existing published world), include `externalRefs` — an array of d-tags that exist in the base world but not in this file. References to them are silently skipped.

### Severity levels

| Level | Meaning | Blocks publish? |
|-------|---------|----------------|
| **error** | Broken references, invalid tag shapes, missing required tags, hash mismatches | Yes |
| **warning** | Structural issues: verb collisions, dangling refs, missing nouns | No (but fix them) |
| **hint** | Discoverability improvements: thin noun aliases, undiscoverable verbs | No |

### Workflow

1. Generate world JSON
2. POST to the validation endpoint
3. Apply each `fix` instruction from the response
4. Re-validate until `"valid": true` (zero errors)
5. Publish

## Walkthrough

Every world should include a `walkthrough` — a sequence of commands that completes the world from start to finish. It proves the world is solvable and catches noun/verb mismatches at authoring time.

### Format

```json
"walkthrough": [
  { "input": "north", "expect": ["Dark Cave"] },
  { "input": "take lantern", "expect": ["Taken"] },
  { "input": "talk hermit", "expect": ["wanderer"] },
  { "input": "1", "expect": ["cave"] },
  { "input": "answer bottle", "expect": ["correct"] }
]
```

### Rules

- Every `input` is a command the player would type
- **Dialogue options are selected by number, not by text** — when options show `[1] Ask about the cave`, the input is `"1"`
- `expect` is an array of substrings that must appear in game output after that command
- Use `reject` instead of `expect` to assert something should NOT appear
- The walkthrough must reach every quest completion and the win state
- Include at least one `examine` for every key entity to validate noun resolution
- Include every puzzle answer and every required verb interaction

## Publishing

### Output format

The JSON file has two top-level keys:

```json
{
  "answers": { "<puzzle-d-tag>": "<plaintext>" },
  "walkthrough": [ ... ],
  "events": [ ... ]
}
```

- `answers` drives NIP-44 encryption and hash verification. The publishing tool strips it before signing — plaintext never reaches the relay.
- `walkthrough` is optional but strongly recommended.
- `events` is the array of kind 30078 events.

For worlds with no NIP-44 sealed events, `answers` can be `{}` or omitted.

### How to publish

Import your JSON into the FOAKLOAR client's builder mode. The tool encrypts NIP-44 content, verifies hashes, strips the `answers` map, signs each event, and publishes to your configured relays.

To update an event, republish with the same `d` tag and pubkey — NOSTR replaceable events mean the relay keeps only the latest version.

**Final deliverable is valid JSON with no JS comments.**

## Sound Scoring

Sound is atmosphere, never information. Never gate a puzzle solution or narrative reveal behind sound alone.

- Score sound against the voice and the win-state feeling, not against the map
- Define sounds as `type: sound` events, reference them by event ref on any other event
- **Layering budget:** 1 ambient + 1 layer per place, 1 layer or effect per entity, 3-4 active max
- Silence is part of the palette — a place with no sound tags inherits a quieter mix
- State-conditional layers: the state element on the `sound` play tag gates the layer automatically

See the [tag reference](./tag-reference.md) for sound event tags and [sample-presets.md](../spec/sample-presets.md) for available sample libraries.

## Consistency Checklist

Before publishing, verify:

- [ ] Every `exit` slot on a place has a portal — or is intentionally dangling
- [ ] Every `requires` references an event that exists and a state that can be set
- [ ] Every `on-complete` and `on-fail` has blank trigger-target `""`
- [ ] Every item the player needs is reachable without needing itself
- [ ] Every NPC has at least one dialogue entry point with no `requires`
- [ ] The win state is reachable by following the critical path
- [ ] Noun tags contain no articles (`the`, `a`, `an`) — the client strips them from input
- [ ] No `verb` tags for built-in commands (`examine`, `take`, `attack`, `talk`, `drop`)
- [ ] Items inside containers are NOT also listed on the place event
- [ ] The voice is consistent across all content fields and transition text
